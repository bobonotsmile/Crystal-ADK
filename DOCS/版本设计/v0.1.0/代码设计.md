# crystal-adk v0.1.0 代码设计

## 1. 目标

`crystal-adk` 的 v0.1.0 不是做一个大而全的 Agent 框架，而是做一个“足够简约但顺手”的 Agent Development Kit。

这一版的核心目标只有两个：

1. 外部项目开发 Agent 时，尽量少自己管理消息数组、tool loop、发送入口。
2. ADK 保持轻量，不耦合 ASP.NET Core Web 宿主模型、不绑定具体业务、不预设复杂工作流。

一句话定义：

> `crystal-adk` 是一个帮助外部项目少管理会话状态、少管理消息协议、少管理工具调用循环的轻量 Agent 开发包。

---

## 2. 开发者使用体验

本节先不讲内部实现，先看我们希望外部项目怎么使用它。

### 2.1 初始化 Host 与配置参数

开发者初始化时，需要配置：

- 系统提示词
- 最大轮次
- provider
- tools

预期使用体验如下：

```csharp
using Crystal.Adk;
using Crystal.Adk.Core;

var tools = new IAgentTool[]
{
    new AnalysisDeviceOverviewTool(deviceService),
    new SearchKnowledgeBaseTool(kbService)
};

var provider = new OpenAiChatProvider(new OpenAiChatProviderOptions
{
    // 鉴权 Key
    ApiKey = Environment.GetEnvironmentVariable("OPENAI_API_KEY") ?? string.Empty,

    // 服务地址
    BaseUrl = "https://api.openai.com/v1",

    // 模型名称
    Model = "gpt-4.1-mini",

    // 单次 HTTP 请求超时（毫秒）
    TimeoutMs = 30000,

    // 采样温度，越低越稳定
    Temperature = 0.2,

    // 核采样阈值
    TopP = 0.9,

    // 最大输出 token 数
    MaxOutputTokens = 4096,

    // 命中停止词时停止输出
    Stop = new[] { "```" },

    // 工具调用策略：auto / none / required
    ToolChoice = "auto",

    // 是否允许一次返回多个工具调用
    ParallelToolCalls = true,

    // 是否开启思考模式
    EnableThinking = false,

    // 厂商扩展字段：统一抽象未覆盖的参数仍可透传
    VendorOptions = new Dictionary<string, object?>
    {
        ["reasoning_effort"] = "medium"
    }
});

var host = new AgentHost(new AgentHostOptions
{
    SystemPrompt = """
    你是设备履历分析助手。
    当需要结构化数据时优先调用工具，不要编造设备状态。
    """,
    MaxRounds = 8
}, provider, tools);
```

这里的重点是：

- 外部项目只提供配置和业务工具
- 外部项目不自己管理 tool loop
- 外部项目不自己拼 message 协议

### 2.2 创建 Session 并发送消息

v0.1.0 预期对外提供 `AgentSession`。

使用体验如下：

```csharp
var session = host.CreateSession();

var result = await session.RunAsync("查询设备 016-108 的履历");

Console.WriteLine(result.FinalMessage);
```

这里的重点是：

- `Session` 持有会话内消息状态
- `RunAsync` 自动追加 `user` 消息
- 内部自动完成 `assistant -> tool -> assistant`
- 外部项目拿到的是最终结果

### 2.3 非流式调用

非流式场景下，开发者只关心最终结果：

```csharp
var result = await session.RunAsync("帮我分析设备 016-108 最近异常");

Console.WriteLine(result.FinalMessage);
Console.WriteLine(result.ToolRuns.Count);
```

这一版需要支持：

- 最终文本
- 工具执行记录
- 当前会话消息列表

### 2.4 流式文本输出

v0.1.0 中，流式能力首先提供“文本流”，用于最直接的打字机式输出体验。

预期使用体验如下：

```csharp
await foreach (var chunk in session.StreamTextAsync("查询设备 016-108 的履历"))
{
    Console.Write(chunk.Text);
}
```

特性：

- 采用sse协议的流式内容输出
- 外部项目看到的是助手文本增量进行消费
- `StreamTextAsync` 是 `RunAsync` 的流式文本版本
- 外部项目不需要自己维护 provider 流式协议
- 工具调用仍然由 ADK 内部处理

禁止：

- 不允许调用Agent的流式事件输出

### 2.5 工具调用体验

业务项目需要定义工具，但不需要关心 tool loop 如何推进。

预期工具写法如下：

```csharp
public sealed class AnalysisDeviceOverviewTool : IAgentTool
{
    public AgentToolDescriptor Descriptor { get; } = new()
    {
        Name = "analysis_device_overview",
        Description = "查询设备分析总览",
        ParametersSchema = new
        {
            type = "object",
            properties = new
            {
                equipNumber = new { type = "string", description = "设备编号" }
            },
            required = new[] { "equipNumber" }
        }
    };

    public async Task<object?> InvokeAsync(Dictionary<string, object?> arguments, CancellationToken cancellationToken)
    {
        var equipNumber = arguments["equipNumber"]?.ToString();
        return await _deviceService.GetOverviewAsync(equipNumber!, cancellationToken);
    }
}
```

这里的重点是：

- 工具只负责业务能力
- 工具不负责消息追加
- 工具不负责调用下一轮模型
- 工具不负责维护会话数组

补充说明：

- 不同厂商的工具调用协议字段存在明显差异，ADK 必须负责适配，而不能把这些协议差异直接暴露给业务工具开发者
- 例如 ARK / ZAI 的 `tool_calls[].function.arguments` 常见为 JSON 字符串，Ollama 的 `arguments` 常见为对象
- 例如 ARK / ZAI 常见存在 `tool_call_id` / `id` 的绑定关系，而 Ollama 这类协议可能需要依赖顺序、补齐内部 callId 或在运行时内部建立映射
- 因此业务工具接口应保持稳定，协议差异由 Provider 适配层和 Agent 运行时共同消化

### 2.6 流式事件输出

除了文本流，v0.1.0 也支持 Agent 视角的流式事件输出。

这里需要先明确一个边界：

- `StreamTextAsync(...)` 面向“只关心文本”的开发者
- `StreamEventsAsync(...)` 面向“需要观察 Agent 运行过程”的开发者

当前阶段，流式事件的核心价值主要来自工具调用场景。

原因是：

- 如果一次请求只是普通文本直答，那么流式价值主要就是文本增量
- 一旦进入 tool loop，开发者往往需要知道“模型何时决定调工具、调了哪个工具、工具什么时候返回、最终回答何时完成”

所以 v0.1.0 的事件流，首先服务于 Agent/tool 主流程，而不是去镜像 provider 的全部原始流式协议。

预期使用体验如下：

```csharp
await foreach (var evt in session.StreamEventsAsync("查询设备 016-108 的履历"))
{
    switch (evt.Kind)
    {
        case AgentEventKinds.RunStarted:
            Console.WriteLine("run started");
            break;

        case AgentEventKinds.ToolCallStarted:
            Console.WriteLine($"calling tool: {evt.ToolName}");
            break;

        case AgentEventKinds.ToolCallCompleted:
            Console.WriteLine($"tool result: {evt.Result}");
            break;

        case AgentEventKinds.TextDelta:
            Console.Write(evt.Text);
            break;

        case AgentEventKinds.FinalAnswer:
            Console.WriteLine();
            Console.WriteLine(evt.Text);
            break;
    }
}
```

这一层里：

1. 事件名称由 `crystal-adk` 定义
   - 事件名称不应直接跟随某个厂商的原始协议字段
   - 事件名称应站在 ADK 的稳定语义层定义
   - 也就是说，开发者看到的是 `RunStarted / ToolCallStarted / ToolCallCompleted / TextDelta / FinalAnswer / RunCompleted / RunFailed`
   - 而不是 `delta`、`choices`、`chunk`、`done` 这类 provider 原始概念

2. 事件内容由 ADK 归一化管理
   - provider 层负责把厂商协议转成 ADK 能理解的中间结果
   - AgentHost / Session 负责把运行过程提升为统一事件
   - 开发者不需要自己解析 SSE、NDJSON 或厂商 chunk 结构

3. 业务项目只消费事件，不定义事件协议
   - 业务项目可以监听事件并决定怎么展示
   - 业务项目不应该定义一套自己的底层事件命名再反向要求 ADK 适配

4. v0.1.0 的事件流范围先收敛在 Agent 行为上
   - 重点覆盖 tool loop 和最终回答
   - 不追求暴露全部 provider 原始细节
   - 这样更利于跨厂商统一，也更不容易在第一版把接口做脏

关于扩展性，这样设计不会妨碍后续扩展，前提是遵守两个原则：

- 第一，v0.1.0 只暴露少量稳定高价值事件，不暴露过多底层细节
- 第二，后续新增事件时采取“只追加、不轻易改语义”的策略

这样未来即使支持更多 provider、更复杂的 agent workflow、多 agent handoff，也仍然可以在现有语义层之上向后兼容扩展。

为避免“事件名只讲原则、不讲落点”，v0.1.0 先将公共事件标识固定为以下几个：

| 事件标识 | 事件含义 | 谁负责定义 | 谁负责产出 | 开发者是否直接消费 | ARK 协议映射来源 |
| --- | --- | --- | --- | --- | --- |
| `run_started` | 一次 Agent 运行开始 | `crystal-adk` | `AgentSession` / `AgentHost` | 是 | 本地运行时事件，不直接来自 ARK 返回 |
| `tool_call_started` | 模型已决定调用某个工具，ADK 准备执行该工具 | `crystal-adk` | `AgentHost` | 是 | 非流式来自 `choices[].message.tool_calls[]`；流式来自 `choices[].delta.tool_calls[]` |
| `tool_call_completed` | 某个工具调用完成并拿到结果 | `crystal-adk` | `AgentHost` | 是 | 本地工具执行结果，不直接来自 ARK 返回 |
| `text_delta` | 助手文本增量 | `crystal-adk` | Provider 适配层上报，`AgentSession` / `AgentHost` 归一化输出 | 是 | `choices[].delta.content` |
| `final_answer` | 一次运行的最终可见回答 | `crystal-adk` | `AgentHost` | 是 | 非流式来自 `choices[].message.content`；流式由全部 `delta.content` 聚合得到 |
| `run_completed` | 一次 Agent 运行成功结束 | `crystal-adk` | `AgentSession` / `AgentHost` | 是 | 本地运行时事件，不直接来自 ARK 返回 |
| `run_failed` | 一次 Agent 运行失败结束 | `crystal-adk` | `AgentSession` / `AgentHost` | 是 | 本地异常边界事件，不直接来自 ARK 返回 |

这张表里最重要的结论有四个：

1. 事件名由 ADK 定，不由厂商定
   - 例如 ARK 给的是 `choices`、`delta`、`finish_reason`、`tool_calls`
   - 这些是协议解析字段，不是外部开发者应该直接面对的 API 事件名
   - 所以 ADK 对外暴露的是 `tool_call_started`、`text_delta` 这类稳定语义事件

2. 不是所有事件都直接来自厂商返回
   - `text_delta` 这类事件和厂商流式协议直接相关
   - `tool_call_started` 是“模型输出 + ADK 识别后”提升出来的事件
   - `tool_call_completed`、`run_completed`、`run_failed` 则是 ADK 本地运行时事件

3. 当前事件流主要围绕 tool loop 展开
   - 对于普通文本直答场景，开发者大多只需要 `text_delta`
   - 一旦进入 `tool_calls`，开发者才明显需要观察 Agent 行为过程
   - 所以你前面说“目前只有工具调用场景才会有流式事件”这个判断，作为 v0.1.0 的重点方向是成立的
   - 更准确地说：v0.1.0 的事件流设计重点主要由工具调用场景驱动

4. 这种定义方式不会妨碍后续扩展
   - 因为现在暴露的是语义层，而不是某家协议的原始字段
   - 后续如果要增加 `reasoning_delta`、`usage_updated`、`handoff_started` 等事件，只要追加即可
   - 只要不随意改变现有事件语义，就能保持向后兼容

如果以你提供的 ARK 协议为例，一次流式 tool calling 过程在 ADK 里的典型映射可以理解为：

1. `StreamEventsAsync(...)` 启动
   - ADK 发出 `run_started`

2. ARK 流式返回 `choices[].delta.content`
   - ADK 发出一个或多个 `text_delta`

3. ARK 返回 `choices[].delta.tool_calls` 或最终 `choices[].message.tool_calls`
   - ADK 识别到要调工具
   - 对每个工具发出 `tool_call_started`

4. ADK 执行工具并得到结果
   - ADK 发出 `tool_call_completed`

5. 后续模型继续生成最终文本
   - ADK 持续发出 `text_delta`

6. 完整回答形成
   - ADK 发出 `final_answer`
   - 随后发出 `run_completed`

如果中途 provider 调用失败、工具执行失败、解析失败或超出轮次：

- ADK 发出 `run_failed`

这就是“事件名称由 `crystal-adk` 定义”在 v0.1.0 里的实际落地方式。

### 2.7 开发者需要拿出 Session 的消息数组

我们已经确认：`Session` 应该管理会话内消息状态，但开发者仍然需要有能力拿到消息数组。

预期使用体验如下：

```csharp
var session = host.CreateSession();

await session.RunAsync("你好");
await session.RunAsync("查询设备 016-108");

var messages = session.Messages;

foreach (var message in messages)
{
    Console.WriteLine($"{message.Role}: {message.Content}");
}
```

这里的设计原则是：

- 会话状态默认由 ADK 管
- 但消息历史不应该被 ADK 藏起来
- 外部项目可以读取消息数组用于调试、日志、持久化、上下文裁剪

因此 v0.1.0 建议：

- `AgentSession` 内部持有消息数组
- 对外暴露 `IReadOnlyList<RuntimeMessage> Messages`
- 暂不允许外部直接随意篡改内部列表

## 3. v0.1.0 推荐的分层

### 3.1 Core 层

Core 层是运行时内核，负责把一次 Agent 交互真正跑起来。

这一层建议包含：

- `RuntimeMessage`
- `RuntimeToolCall`
- `AgentToolDescriptor`
- `AgentRunResult`
- `AgentHost`
- `IAgentTool`
- `ToolHub`
- `IChatProvider`

职责：

- 统一消息协议
- 统一工具协议
- 负责 tool loop
- 负责 provider 调用与工具回填

### 3.2 Session 层

Session 层是外部开发者最常接触的层。

这一层建议包含：

- `AgentSession`
- `RunAsync`
- `StreamTextAsync`
- `StreamEventsAsync`
- `Messages`
- `Reset`

职责：

- 持有会话内消息数组
- 自动追加 `user` 消息
- 调用 `AgentHost`
- 对外暴露只读消息历史

一句话：

- Core 负责“怎么跑”
- Session 负责“怎么用”

## 4. 项目代码目录设计

为了让 `crystal-adk` 在 v0.1.0 阶段保持可维护、可扩展、不过度拆分，项目代码目录建议采用“按职责分区”的方式组织，而不是按厂商、按 demo、按概念随意平铺。

推荐目录结构如下：

```text
crystal-adk/
├─ Core/
│  ├─ Messages/
│  ├─ Tools/
│  ├─ Events/
│  ├─ Results/
│  ├─ AgentHost.cs
│  └─ AgentHostOptions.cs
├─ Session/
│  ├─ AgentSession.cs
│  ├─ TextChunk.cs
│  └─ AgentEvent.cs
├─ Providers/
│  ├─ Abstractions/
│  ├─ Ark/
│  ├─ Ollama/
│  └─ Zai/
├─ Serialization/
│  ├─ MessageMappers/
│  ├─ ToolCallMappers/
│  └─ Json/
├─ Internal/
│  ├─ ToolLoop/
│  ├─ Streaming/
│  └─ Utilities/
├─ DOCS/
└─ Crystal.Adk.csproj
```

这一版不要求第一次就把目录完全拆到最细，但设计目标应朝这个方向收敛。

### 4.1 `Core/`

`Core/` 放运行时最核心、最稳定、最不应依赖具体厂商和宿主形式的类型。

建议放：

- 消息协议
- 工具协议
- 事件协议
- 运行结果协议
- `AgentHost`
- `AgentHostOptions`

这层的要求是：

- 不依赖具体 provider 实现
- 不依赖业务工具
- 不依赖 ASP.NET Core Web 宿主模型

### 4.2 `Session/`

`Session/` 放面向外部开发者的使用入口。

建议放：

- `AgentSession`
- `TextChunk`
- `AgentEvent`
- 与 Session 生命周期相关的轻量类型

这层的职责是：

- 管理会话内消息状态
- 暴露 `RunAsync`
- 暴露 `StreamTextAsync`
- 暴露 `StreamEventsAsync`
- 暴露 `Messages`

也就是说，`Session/` 更偏“外部 API 体验层”。

### 4.3 `Providers/`

`Providers/` 放各厂商 Chat API 的适配实现。

建议结构为：

- `Providers/Abstractions/`
  - `IChatProvider`
  - 公共 provider 结果模型
- `Providers/Ark/`
  - ARK 请求/响应适配
- `Providers/Ollama/`
  - Ollama 请求/响应适配
- `Providers/Zai/`
  - ZAI 请求/响应适配

这里要特别强调：

- 厂商协议差异主要由这一层承担
- 例如 tool call 的 `arguments` 是字符串还是对象
- 例如 tool 结果绑定是 `tool_call_id` 还是需要内部补齐 callId
- 例如流式返回是 SSE chunk 还是 NDJSON

这些差异不应该泄漏到 `AgentSession` 和业务工具接口。

### 4.4 `Serialization/`

`Serialization/` 放协议映射和 JSON 相关代码。

建议放：

- 消息数组到厂商请求体的映射
- 厂商响应到 ADK 内部模型的映射
- tool call 参数解析
- JSON 容错解析

为什么单独放这一层：

- Provider 实现里最容易膨胀的部分就是序列化和字段映射
- 如果不抽开，`ArkProvider` / `OllamaProvider` 这类文件会迅速变得很长

### 4.5 `Internal/`

`Internal/` 放不打算作为公共 API 暴露，但运行时内部又需要复用的实现细节。

建议放：

- tool loop 细节
- 流式拼接器
- 内部辅助方法
- 事件分发器

这层的原则是：

- 可以变化快
- 可以重构
- 不应作为外部项目依赖的稳定入口

### 4.6 v0.1.0 阶段的目录克制原则

虽然上面给出了较完整的目标目录，但 v0.1.0 仍要保持克制：

1. 不必一开始就拆成大量空目录
2. 可以先按较粗颗粒落地
3. 只有当某个文件群真正开始膨胀时，再下沉成子目录

因此 v0.1.0 允许的实际落地结构可以更简化，例如：

```text
crystal-adk/
├─ Core/
├─ Session/
├─ Providers/
├─ Internal/
├─ DOCS/
└─ Crystal.Adk.csproj
```

然后随着实现推进，再把：

- `Messages/`
- `Tools/`
- `Events/`
- `Serialization/`

继续拆细。

### 4.7 目录设计的边界结论

关于项目代码目录，v0.1.0 的结论是：

1. 目录按职责划分，而不是按功能演示划分
2. `Core` 与 `Session` 必须分开
3. `Providers` 必须单独成区，承担厂商协议差异
4. 协议映射与 JSON 解析逻辑不应长期堆在单个 Provider 文件里
5. `Internal` 可以存在，但不应成为对外 API 的主要入口
6. 目录结构要为后续扩展留空间，但不能为了“未来可能会用”而过度拆分

## 5. 哪些由 ADK 管

### 4.1 ADK 负责管理的内容

v0.1.0 中，以下内容应该由 ADK 负责：

1. 消息协议
   - `system`
   - `user`
   - `assistant`
   - `tool`

2. 会话内消息状态
   - 持有消息列表
   - 自动追加消息
   - 支持读取历史
   - 支持清空会话

3. Agent 执行主流程
   - 请求 provider
   - 识别 `tool_calls`
   - 调用 tool
   - 把 tool 结果回填到消息中
   - 继续下一轮直到结束

4. Tool 注册与分发
   - 注册工具
   - 根据名字查找工具
   - 执行工具

5. 统一运行结果
   - 最终文本
   - 工具执行记录
   - 当前消息历史

6. 流式/非流式统一调用入口
   - `RunAsync`
   - `StreamTextAsync`
   - `StreamEventsAsync`

7. 基础保护逻辑
   - 最大轮次
   - 重复 tool call 防护
   - 基础异常边界

### 4.2 ADK 管理这些内容的原因

这些内容有一个共同特点：

- 每个 Agent 项目都要写
- 重复度高
- 写错后很难排查
- 不属于具体业务

所以它们应该被 ADK 吞进去，减少外部项目的心智负担。

## 6. 哪些暴露给外部项目

### 5.1 外部项目负责的内容

v0.1.0 中，以下内容应由外部项目负责：

1. 业务工具
   - 例如设备履历查询
   - 例如知识库搜索
   - 例如工单创建

2. 业务提示词
   - 每个 Agent 的系统提示词
   - 每个 Agent 的回复风格与约束

3. provider 选择与配置
   - API Key
   - Model
   - BaseUrl
   - Timeout
   - 通用推理参数
   - 厂商扩展参数

4. 宿主方式
   - 控制台
   - ASP.NET
   - Windows Service
   - 后台任务

5. 业务服务与数据源
   - 数据库
   - HTTP API
   - 领域服务
   - Repository / Query Service

### 5.2 为什么这些不应进 ADK

这些内容带有明显业务或部署环境特征：

- 并不是每个 Agent 都一样
- 变化快
- 一旦塞进 ADK，就会快速污染边界

所以 v0.1.0 要明确克制：

- ADK 不内置设备履历工具
- ADK 不内置 SQL
- ADK 不内置业务 handler
- ADK 不替用户决定模型与提示词

## 7. v0.1.0 不做什么

为了避免过度设计，v0.1.0 明确不做以下能力。

### 6.1 不做复杂 DSL

不做类似下面这种链式框架表达：

```csharp
prompt | model | parser | tools | memory
```

原因：

- 现在还没有稳定的使用习惯
- 抽象过早，容易把简单问题复杂化

### 6.2 不做复杂 Agent 类型体系

不预置：

- `BaseAgent`
- `PlannerAgent`
- `WorkflowAgent`
- `ReActAgent`

原因：

- 当前目标是“简约可用”
- 不是先做一个大框架分类树

### 6.3 不做插件系统

不做：

- 程序集动态加载 tool
- 插件市场
- 自动扫描插件包

原因：

- 这不是当前痛点
- 容易显著增加复杂度

### 6.4 不做持久化 Session Store

v0.1.0 中，`Session` 只管理进程内内存状态，不负责：

- Redis 持久化
- 数据库存档
- 自动恢复会话

原因：

- 当前先把内存会话模型跑顺
- 持久化应该是后续扩展点，而不是 v0.1.0 的默认能力

### 6.5 不做上下文压缩和摘要

暂不做：

- 自动裁剪消息
- 自动总结历史
- 自动窗口压缩

原因：

- 这会引入更多策略问题
- 现在先保证基础会话模型足够清晰

### 6.6 不耦合 ASP.NET Core Web 宿主模型

`crystal-adk` 在 v0.1.0 中是一个基于 .NET 的普通类库，可以运行在 `net6.0` 上，但不直接耦合 ASP.NET Core Web 宿主模型。

这里需要明确区分两件事：

- `net6.0` 是目标运行时版本
- ASP.NET Core 是 Web 应用宿主与框架模型

因此，v0.1.0 不直接依赖以下 Web 宿主相关类型与模式：

- `HttpContext`
- `WebApplication`
- `MapPost`
- `IResult`
- `SSE`

原因：

- ADK 应该是纯运行时 + Session 体验层
- Web 只是宿主形式之一

### 6.7 v0.1.0 事件流范围控制

v0.1.0 对外提供 `StreamEventsAsync(...)`，但事件流范围会刻意收敛。

本期同时做：

- `RunAsync(...)`：非流式，返回最终结果
- `StreamTextAsync(...)`：文本流，只暴露助手文本增量
- `StreamEventsAsync(...)`：Agent 事件流，暴露稳定语义事件

本期事件流重点覆盖：

- `RunStarted`
- `ToolCallStarted`
- `ToolCallCompleted`
- `TextDelta`
- `FinalAnswer`
- `RunCompleted`
- `RunFailed`

本期仍然不做：

- provider 原始 chunk 全量透传
- 厂商级别的底层事件直出
- 过细的内部调度事件

原因：

- 事件命名应该由 ADK 语义层定义，而不是绑定某家协议
- 不同厂商的流式协议差异较大，先做稳定语义层更合理
- v0.1.0 需要的是“好用且可扩展”的事件流，而不是“最底层、最原始”的事件流

## 8. Session 设计结论

关于会话与消息状态，我们在本轮讨论中的结论如下：

1. `Session` 应该管理“会话内消息状态”
2. `Session` 是外部开发者的主要使用入口
3. `Session` 应提供 `RunAsync`
4. `Session` 在 v0.1.0 应提供文本流接口 `StreamTextAsync`
5. `Session` 在 v0.1.0 也应提供 Agent 事件流接口 `StreamEventsAsync`
6. `Session` 必须允许开发者拿出当前对话数组
7. 但对外应优先暴露只读消息列表，而不是可随意修改的内部集合

因此 v0.1.0 推荐的接口方向是：

```csharp
public sealed class AgentSession
{
    public IReadOnlyList<RuntimeMessage> Messages { get; }

    public Task<AgentRunResult> RunAsync(string userInput, CancellationToken cancellationToken = default);

    public IAsyncEnumerable<TextChunk> StreamTextAsync(string userInput, CancellationToken cancellationToken = default);

    public IAsyncEnumerable<AgentEvent> StreamEventsAsync(string userInput, CancellationToken cancellationToken = default);

    public void Reset();
}
```

这个接口表达的核心意图是：

- 用户只管“发一句话”
- ADK 负责把这句话推进成完整的一轮 Agent 交互
- 但用户仍可观察会话状态

## 9. Provider 参数设计结论

关于 Chat Provider 的参数设计，v0.1.0 的结论是：

1. 不能只支持 `ApiKey / Model / BaseUrl / TimeoutMs` 这几个最小字段
2. 也不能直接把每个厂商的全部字段平铺到一个超大 Options 类里
3. 应采用“扁平共性字段 + 厂商扩展字段”的双层模型

### 8.1 为什么采用“扁平共性字段 + 厂商扩展字段”

当前计划兼容的三家协议包括：

- ARK
- Ollama
- ZAI

它们的共性非常明显：

- `model`
- `messages`
- `stream`
- `tools`
- `tool_choice`
- `temperature`
- `top_p`
- `max_tokens` 或等价字段
- `stop`

但差异也很明显：

- ARK 有 `thinking.type`、`reasoning_effort`、`parallel_tool_calls`
- Ollama 有 `options`、`think`、`keep_alive`、`format`
- ZAI 有 `thinking.clear_thinking`、`tool_stream`、`do_sample`、`request_id`、`user_id`

因此如果只保留极少共性字段，开发者会觉得不够用。

但如果把所有厂商字段都硬塞到一个统一类里：

- API 会很臃肿
- 含义会变模糊
- 很多字段在其它厂商上根本不成立

### 8.2 推荐的 Options 结构

v0.1.0 不再额外嵌套 `ChatRequestOptions`，而是将“连接参数”和“通用请求参数”放在同一个 `ProviderOptions` 上，保持开发者配置体验足够直接。

结构上分为两类：

1. 扁平共性字段
   - `ApiKey`
   - `Model`
   - `BaseUrl`
   - `TimeoutMs`
   - `Temperature`
   - `TopP`
   - `MaxOutputTokens`
   - `Stop`
   - `ToolChoice`
   - `ParallelToolCalls`
   - `ResponseFormat`
   - `EnableThinking`

2. 厂商扩展字段
   - `VendorOptions`
   - 类型建议为 `Dictionary<string, object?>`

示意如下：

```csharp
public sealed class OpenAiChatProviderOptions
{
    public string ApiKey { get; set; } = string.Empty;
    public string Model { get; set; } = string.Empty;
    public string BaseUrl { get; set; } = string.Empty;
    public int TimeoutMs { get; set; } = 30000;

    public double? Temperature { get; set; }
    public double? TopP { get; set; }
    public int? MaxOutputTokens { get; set; }
    public IReadOnlyList<string>? Stop { get; set; }
    public string? ToolChoice { get; set; }
    public bool? ParallelToolCalls { get; set; }
    public string? ResponseFormat { get; set; }
    public bool? EnableThinking { get; set; }

    public Dictionary<string, object?> VendorOptions { get; set; } = new();
}
```

这样设计的原因是：

- 外部开发者配置时更顺手
- v0.1.0 更符合“少一层抽象”的目标
- 大部分使用者会把这些都理解为“provider 配置”，没必要强行再套一层

### 8.3 字段映射原则

v0.1.0 建议字段映射遵循以下原则：

1. 优先支持三家都有或语义接近的字段
2. 对命名不同但语义接近的字段做统一抽象
3. 无法统一的字段放入 `VendorOptions`
4. Provider 在发 HTTP JSON 时负责把通用字段转换成厂商字段

例如：

- 通用 `MaxOutputTokens`
  - ARK -> `max_tokens` 或 `max_completion_tokens`
  - Ollama -> `options.num_predict`
  - ZAI -> `max_tokens`

- 通用 `EnableThinking`
  - ARK -> `thinking.type = enabled/disabled`
  - Ollama -> `think = true/false` 或级别字符串
  - ZAI -> `thinking.type = enabled/disabled`

### 8.4 v0.1.0 的克制点

这一版虽然支持 `VendorOptions`，但仍要保持克制：

- 不追求一次性覆盖所有厂商全部字段
- 不做复杂 schema 校验器
- 不做厂商协议自动发现
- 不做完整多模态统一抽象

本期重点仍然是：

- 多轮 messages
- Function Calling
- 非流式 `RunAsync`
- 文本流 `StreamTextAsync`
- Agent 事件流 `StreamEventsAsync`

---

## 10. v0.1.0 的最小成功标准

如果这一版设计落地成功，那么外部项目开发一个新 Agent 时，至少可以做到：

1. 不自己维护 `messages` 数组追加逻辑
2. 不自己维护 tool loop
3. 不自己手动把 tool result 回填到消息中
4. 可以直接 `RunAsync(...)`
5. 可以切换成文本流 `StreamTextAsync(...)`
6. 在需要时可以切换成 Agent 事件流 `StreamEventsAsync(...)`
7. 可以随时读取当前会话消息历史
8. 业务项目只需关心工具、提示词、provider 配置与宿主方式

这就是 `crystal-adk v0.1.0` 设计的目标边界。

# crystal-adk v0.1.0 代码设计

## 1. 目标

`crystal-adk` 的 v0.1.0 不是做一个大而全的 Agent 框架，而是做一个“足够简约但顺手”的 Agent Development Kit。

这一版的核心目标只有两个：

1. 外部项目开发 Agent 时，尽量少自己管理消息数组、tool loop、发送入口。
2. ADK 保持轻量，不耦合 ASP.NET Core Web 宿主模型、不绑定具体业务、不预设复杂工作流。

一句话定义：

> `crystal-adk` 是一个帮助外部项目少管理会话状态、少管理消息协议、少管理工具调用循环的轻量 Agent 开发包。

---

## 2. 开发者使用体验

本节先不讲内部实现，先看我们希望外部项目怎么使用它。

### 2.1 初始化 Host 与配置参数

开发者初始化时，需要配置：

- 系统提示词
- 最大轮次
- provider
- tools

预期使用体验如下：

```csharp
using Crystal.Adk;
using Crystal.Adk.Core;

var tools = new IAgentTool[]
{
    new AnalysisDeviceOverviewTool(deviceService),
    new SearchKnowledgeBaseTool(kbService)
};

var provider = new OpenAiChatProvider(new OpenAiChatProviderOptions
{
    // 鉴权 Key
    ApiKey = Environment.GetEnvironmentVariable("OPENAI_API_KEY") ?? string.Empty,

    // 服务地址
    BaseUrl = "https://api.openai.com/v1",

    // 模型名称
    Model = "gpt-4.1-mini",

    // 单次 HTTP 请求超时（毫秒）
    TimeoutMs = 30000,

    // 采样温度，越低越稳定
    Temperature = 0.2,

    // 核采样阈值
    TopP = 0.9,

    // 最大输出 token 数
    MaxOutputTokens = 4096,

    // 命中停止词时停止输出
    Stop = new[] { "```" },

    // 工具调用策略：auto / none / required
    ToolChoice = "auto",

    // 是否允许一次返回多个工具调用
    ParallelToolCalls = true,

    // 是否开启思考模式
    EnableThinking = false,

    // 厂商扩展字段：统一抽象未覆盖的参数仍可透传
    VendorOptions = new Dictionary<string, object?>
    {
        ["reasoning_effort"] = "medium"
    }
});

var host = new AgentHost(new AgentHostOptions
{
    SystemPrompt = """
    你是设备履历分析助手。
    当需要结构化数据时优先调用工具，不要编造设备状态。
    """,
    MaxRounds = 8
}, provider, tools);
```

这里的重点是：

- 外部项目只提供配置和业务工具
- 外部项目不自己管理 tool loop
- 外部项目不自己拼 message 协议

### 2.2 创建 Session 并发送消息

v0.1.0 预期对外提供 `AgentSession`。

使用体验如下：

```csharp
var session = host.CreateSession();

var result = await session.RunAsync("查询设备 016-108 的履历");

Console.WriteLine(result.FinalMessage);
```

这里的重点是：

- `Session` 持有会话内消息状态
- `RunAsync` 自动追加 `user` 消息
- 内部自动完成 `assistant -> tool -> assistant`
- 外部项目拿到的是最终结果

### 2.3 非流式调用

非流式场景下，开发者只关心最终结果：

```csharp
var result = await session.RunAsync("帮我分析设备 016-108 最近异常");

Console.WriteLine(result.FinalMessage);
Console.WriteLine(result.ToolRuns.Count);
```

这一版需要支持：

- 最终文本
- 工具执行记录
- 当前会话消息列表

### 2.4 流式文本输出

v0.1.0 中，流式能力首先提供“文本流”，用于最直接的打字机式输出体验。

预期使用体验如下：

```csharp
await foreach (var chunk in session.StreamTextAsync("查询设备 016-108 的履历"))
{
    Console.Write(chunk.Text);
}
```

特性：

- 采用sse协议的流式内容输出
- 外部项目看到的是助手文本增量进行消费
- `StreamTextAsync` 是 `RunAsync` 的流式文本版本
- 外部项目不需要自己维护 provider 流式协议
- 工具调用仍然由 ADK 内部处理

禁止：

- 不允许调用Agent的流式事件输出

### 2.5 工具调用体验

业务项目需要定义工具，但不需要关心 tool loop 如何推进。

预期工具写法如下：

```csharp
public sealed class AnalysisDeviceOverviewTool : IAgentTool
{
    public AgentToolDescriptor Descriptor { get; } = new()
    {
        Name = "analysis_device_overview",
        Description = "查询设备分析总览",
        ParametersSchema = new
        {
            type = "object",
            properties = new
            {
                equipNumber = new { type = "string", description = "设备编号" }
            },
            required = new[] { "equipNumber" }
        }
    };

    public async Task<object?> InvokeAsync(Dictionary<string, object?> arguments, CancellationToken cancellationToken)
    {
        var equipNumber = arguments["equipNumber"]?.ToString();
        return await _deviceService.GetOverviewAsync(equipNumber!, cancellationToken);
    }
}
```

这里的重点是：

- 工具只负责业务能力
- 工具不负责消息追加
- 工具不负责调用下一轮模型
- 工具不负责维护会话数组

补充说明：

- 不同厂商的工具调用协议字段存在明显差异，ADK 必须负责适配，而不能把这些协议差异直接暴露给业务工具开发者
- 例如 ARK / ZAI 的 `tool_calls[].function.arguments` 常见为 JSON 字符串，Ollama 的 `arguments` 常见为对象
- 例如 ARK / ZAI 常见存在 `tool_call_id` / `id` 的绑定关系，而 Ollama 这类协议可能需要依赖顺序、补齐内部 callId 或在运行时内部建立映射
- 因此业务工具接口应保持稳定，协议差异由 Provider 适配层和 Agent 运行时共同消化

### 2.6 流式事件输出

除了文本流，v0.1.0 也支持 Agent 视角的流式事件输出。

这里需要先明确一个边界：

- `StreamTextAsync(...)` 面向“只关心文本”的开发者
- `StreamEventsAsync(...)` 面向“需要观察 Agent 运行过程”的开发者

当前阶段，流式事件的核心价值主要来自工具调用场景。

原因是：

- 如果一次请求只是普通文本直答，那么流式价值主要就是文本增量
- 一旦进入 tool loop，开发者往往需要知道“模型何时决定调工具、调了哪个工具、工具什么时候返回、最终回答何时完成”

所以 v0.1.0 的事件流，首先服务于 Agent/tool 主流程，而不是去镜像 provider 的全部原始流式协议。

预期使用体验如下：

```csharp
await foreach (var evt in session.StreamEventsAsync("查询设备 016-108 的履历"))
{
    switch (evt.Kind)
    {
        case AgentEventKinds.RunStarted:
            Console.WriteLine("run started");
            break;

        case AgentEventKinds.ToolCallStarted:
            Console.WriteLine($"calling tool: {evt.ToolName}");
            break;

        case AgentEventKinds.ToolCallCompleted:
            Console.WriteLine($"tool result: {evt.Result}");
            break;

        case AgentEventKinds.TextDelta:
            Console.Write(evt.Text);
            break;

        case AgentEventKinds.FinalAnswer:
            Console.WriteLine();
            Console.WriteLine(evt.Text);
            break;
    }
}
```

这一层里：

1. 事件名称由 `crystal-adk` 定义
   - 事件名称不应直接跟随某个厂商的原始协议字段
   - 事件名称应站在 ADK 的稳定语义层定义
   - 也就是说，开发者看到的是 `RunStarted / ToolCallStarted / ToolCallCompleted / TextDelta / FinalAnswer / RunCompleted / RunFailed`
   - 而不是 `delta`、`choices`、`chunk`、`done` 这类 provider 原始概念

2. 事件内容由 ADK 归一化管理
   - provider 层负责把厂商协议转成 ADK 能理解的中间结果
   - AgentHost / Session 负责把运行过程提升为统一事件
   - 开发者不需要自己解析 SSE、NDJSON 或厂商 chunk 结构

3. 业务项目只消费事件，不定义事件协议
   - 业务项目可以监听事件并决定怎么展示
   - 业务项目不应该定义一套自己的底层事件命名再反向要求 ADK 适配

4. v0.1.0 的事件流范围先收敛在 Agent 行为上
   - 重点覆盖 tool loop 和最终回答
   - 不追求暴露全部 provider 原始细节
   - 这样更利于跨厂商统一，也更不容易在第一版把接口做脏

关于扩展性，这样设计不会妨碍后续扩展，前提是遵守两个原则：

- 第一，v0.1.0 只暴露少量稳定高价值事件，不暴露过多底层细节
- 第二，后续新增事件时采取“只追加、不轻易改语义”的策略

这样未来即使支持更多 provider、更复杂的 agent workflow、多 agent handoff，也仍然可以在现有语义层之上向后兼容扩展。

为避免“事件名只讲原则、不讲落点”，v0.1.0 先将公共事件标识固定为以下几个：

| 事件标识 | 事件含义 | 谁负责定义 | 谁负责产出 | 开发者是否直接消费 | ARK 协议映射来源 |
| --- | --- | --- | --- | --- | --- |
| `run_started` | 一次 Agent 运行开始 | `crystal-adk` | `AgentSession` / `AgentHost` | 是 | 本地运行时事件，不直接来自 ARK 返回 |
| `tool_call_started` | 模型已决定调用某个工具，ADK 准备执行该工具 | `crystal-adk` | `AgentHost` | 是 | 非流式来自 `choices[].message.tool_calls[]`；流式来自 `choices[].delta.tool_calls[]` |
| `tool_call_completed` | 某个工具调用完成并拿到结果 | `crystal-adk` | `AgentHost` | 是 | 本地工具执行结果，不直接来自 ARK 返回 |
| `text_delta` | 助手文本增量 | `crystal-adk` | Provider 适配层上报，`AgentSession` / `AgentHost` 归一化输出 | 是 | `choices[].delta.content` |
| `final_answer` | 一次运行的最终可见回答 | `crystal-adk` | `AgentHost` | 是 | 非流式来自 `choices[].message.content`；流式由全部 `delta.content` 聚合得到 |
| `run_completed` | 一次 Agent 运行成功结束 | `crystal-adk` | `AgentSession` / `AgentHost` | 是 | 本地运行时事件，不直接来自 ARK 返回 |
| `run_failed` | 一次 Agent 运行失败结束 | `crystal-adk` | `AgentSession` / `AgentHost` | 是 | 本地异常边界事件，不直接来自 ARK 返回 |

这张表里最重要的结论有四个：

1. 事件名由 ADK 定，不由厂商定
   - 例如 ARK 给的是 `choices`、`delta`、`finish_reason`、`tool_calls`
   - 这些是协议解析字段，不是外部开发者应该直接面对的 API 事件名
   - 所以 ADK 对外暴露的是 `tool_call_started`、`text_delta` 这类稳定语义事件

2. 不是所有事件都直接来自厂商返回
   - `text_delta` 这类事件和厂商流式协议直接相关
   - `tool_call_started` 是“模型输出 + ADK 识别后”提升出来的事件
   - `tool_call_completed`、`run_completed`、`run_failed` 则是 ADK 本地运行时事件

3. 当前事件流主要围绕 tool loop 展开
   - 对于普通文本直答场景，开发者大多只需要 `text_delta`
   - 一旦进入 `tool_calls`，开发者才明显需要观察 Agent 行为过程
   - 所以你前面说“目前只有工具调用场景才会有流式事件”这个判断，作为 v0.1.0 的重点方向是成立的
   - 更准确地说：v0.1.0 的事件流设计重点主要由工具调用场景驱动

4. 这种定义方式不会妨碍后续扩展
   - 因为现在暴露的是语义层，而不是某家协议的原始字段
   - 后续如果要增加 `reasoning_delta`、`usage_updated`、`handoff_started` 等事件，只要追加即可
   - 只要不随意改变现有事件语义，就能保持向后兼容

如果以你提供的 ARK 协议为例，一次流式 tool calling 过程在 ADK 里的典型映射可以理解为：

1. `StreamEventsAsync(...)` 启动
   - ADK 发出 `run_started`

2. ARK 流式返回 `choices[].delta.content`
   - ADK 发出一个或多个 `text_delta`

3. ARK 返回 `choices[].delta.tool_calls` 或最终 `choices[].message.tool_calls`
   - ADK 识别到要调工具
   - 对每个工具发出 `tool_call_started`

4. ADK 执行工具并得到结果
   - ADK 发出 `tool_call_completed`

5. 后续模型继续生成最终文本
   - ADK 持续发出 `text_delta`

6. 完整回答形成
   - ADK 发出 `final_answer`
   - 随后发出 `run_completed`

如果中途 provider 调用失败、工具执行失败、解析失败或超出轮次：

- ADK 发出 `run_failed`

这就是“事件名称由 `crystal-adk` 定义”在 v0.1.0 里的实际落地方式。

### 2.7 开发者需要拿出 Session 的消息数组

我们已经确认：`Session` 应该管理会话内消息状态，但开发者仍然需要有能力拿到消息数组。

预期使用体验如下：

```csharp
var session = host.CreateSession();

await session.RunAsync("你好");
await session.RunAsync("查询设备 016-108");

var messages = session.Messages;

foreach (var message in messages)
{
    Console.WriteLine($"{message.Role}: {message.Content}");
}
```

这里的设计原则是：

- 会话状态默认由 ADK 管
- 但消息历史不应该被 ADK 藏起来
- 外部项目可以读取消息数组用于调试、日志、持久化、上下文裁剪

因此 v0.1.0 建议：

- `AgentSession` 内部持有消息数组
- 对外暴露 `IReadOnlyList<RuntimeMessage> Messages`
- 暂不允许外部直接随意篡改内部列表

## 3. v0.1.0 推荐的分层

### 3.1 Core 层

Core 层是运行时内核，负责把一次 Agent 交互真正跑起来。

这一层建议包含：

- `RuntimeMessage`
- `RuntimeToolCall`
- `AgentToolDescriptor`
- `AgentRunResult`
- `AgentHost`
- `IAgentTool`
- `ToolHub`
- `IChatProvider`

职责：

- 统一消息协议
- 统一工具协议
- 负责 tool loop
- 负责 provider 调用与工具回填

### 3.2 Session 层

Session 层是外部开发者最常接触的层。

这一层建议包含：

- `AgentSession`
- `RunAsync`
- `StreamTextAsync`
- `StreamEventsAsync`
- `Messages`
- `Reset`

职责：

- 持有会话内消息数组
- 自动追加 `user` 消息
- 调用 `AgentHost`
- 对外暴露只读消息历史

一句话：

- Core 负责“怎么跑”
- Session 负责“怎么用”

## 4. 项目代码目录设计

为了让 `crystal-adk` 在 v0.1.0 阶段保持可维护、可扩展、不过度拆分，项目代码目录建议采用“按职责分区”的方式组织，而不是按厂商、按 demo、按概念随意平铺。

推荐目录结构如下：

```text
crystal-adk/
├─ Core/
│  ├─ Messages/
│  ├─ Tools/
│  ├─ Events/
│  ├─ Results/
│  ├─ AgentHost.cs
│  └─ AgentHostOptions.cs
├─ Session/
│  ├─ AgentSession.cs
│  ├─ TextChunk.cs
│  └─ AgentEvent.cs
├─ Providers/
│  ├─ Abstractions/
│  ├─ Ark/
│  ├─ Ollama/
│  └─ Zai/
├─ Serialization/
│  ├─ MessageMappers/
│  ├─ ToolCallMappers/
│  └─ Json/
├─ Internal/
│  ├─ ToolLoop/
│  ├─ Streaming/
│  └─ Utilities/
├─ DOCS/
└─ Crystal.Adk.csproj
```

这一版不要求第一次就把目录完全拆到最细，但设计目标应朝这个方向收敛。

### 4.1 `Core/`

`Core/` 放运行时最核心、最稳定、最不应依赖具体厂商和宿主形式的类型。

建议放：

- 消息协议
- 工具协议
- 事件协议
- 运行结果协议
- `AgentHost`
- `AgentHostOptions`

这层的要求是：

- 不依赖具体 provider 实现
- 不依赖业务工具
- 不依赖 ASP.NET Core Web 宿主模型

### 4.2 `Session/`

`Session/` 放面向外部开发者的使用入口。

建议放：

- `AgentSession`
- `TextChunk`
- `AgentEvent`
- 与 Session 生命周期相关的轻量类型

这层的职责是：

- 管理会话内消息状态
- 暴露 `RunAsync`
- 暴露 `StreamTextAsync`
- 暴露 `StreamEventsAsync`
- 暴露 `Messages`

也就是说，`Session/` 更偏“外部 API 体验层”。

### 4.3 `Providers/`

`Providers/` 放各厂商 Chat API 的适配实现。

建议结构为：

- `Providers/Abstractions/`
  - `IChatProvider`
  - 公共 provider 结果模型
- `Providers/Ark/`
  - ARK 请求/响应适配
- `Providers/Ollama/`
  - Ollama 请求/响应适配
- `Providers/Zai/`
  - ZAI 请求/响应适配

这里要特别强调：

- 厂商协议差异主要由这一层承担
- 例如 tool call 的 `arguments` 是字符串还是对象
- 例如 tool 结果绑定是 `tool_call_id` 还是需要内部补齐 callId
- 例如流式返回是 SSE chunk 还是 NDJSON

这些差异不应该泄漏到 `AgentSession` 和业务工具接口。

### 4.4 `Serialization/`

`Serialization/` 放协议映射和 JSON 相关代码。

建议放：

- 消息数组到厂商请求体的映射
- 厂商响应到 ADK 内部模型的映射
- tool call 参数解析
- JSON 容错解析

为什么单独放这一层：

- Provider 实现里最容易膨胀的部分就是序列化和字段映射
- 如果不抽开，`ArkProvider` / `OllamaProvider` 这类文件会迅速变得很长

### 4.5 `Internal/`

`Internal/` 放不打算作为公共 API 暴露，但运行时内部又需要复用的实现细节。

建议放：

- tool loop 细节
- 流式拼接器
- 内部辅助方法
- 事件分发器

这层的原则是：

- 可以变化快
- 可以重构
- 不应作为外部项目依赖的稳定入口

### 4.6 v0.1.0 阶段的目录克制原则

虽然上面给出了较完整的目标目录，但 v0.1.0 仍要保持克制：

1. 不必一开始就拆成大量空目录
2. 可以先按较粗颗粒落地
3. 只有当某个文件群真正开始膨胀时，再下沉成子目录

因此 v0.1.0 允许的实际落地结构可以更简化，例如：

```text
crystal-adk/
├─ Core/
├─ Session/
├─ Providers/
├─ Internal/
├─ DOCS/
└─ Crystal.Adk.csproj
```

然后随着实现推进，再把：

- `Messages/`
- `Tools/`
- `Events/`
- `Serialization/`

继续拆细。

### 4.7 目录设计的边界结论

关于项目代码目录，v0.1.0 的结论是：

1. 目录按职责划分，而不是按功能演示划分
2. `Core` 与 `Session` 必须分开
3. `Providers` 必须单独成区，承担厂商协议差异
4. 协议映射与 JSON 解析逻辑不应长期堆在单个 Provider 文件里
5. `Internal` 可以存在，但不应成为对外 API 的主要入口
6. 目录结构要为后续扩展留空间，但不能为了“未来可能会用”而过度拆分

## 5. 哪些由 ADK 管

### 4.1 ADK 负责管理的内容

v0.1.0 中，以下内容应该由 ADK 负责：

1. 消息协议
   - `system`
   - `user`
   - `assistant`
   - `tool`

2. 会话内消息状态
   - 持有消息列表
   - 自动追加消息
   - 支持读取历史
   - 支持清空会话

3. Agent 执行主流程
   - 请求 provider
   - 识别 `tool_calls`
   - 调用 tool
   - 把 tool 结果回填到消息中
   - 继续下一轮直到结束

4. Tool 注册与分发
   - 注册工具
   - 根据名字查找工具
   - 执行工具

5. 统一运行结果
   - 最终文本
   - 工具执行记录
   - 当前消息历史

6. 流式/非流式统一调用入口
   - `RunAsync`
   - `StreamTextAsync`
   - `StreamEventsAsync`

7. 基础保护逻辑
   - 最大轮次
   - 重复 tool call 防护
   - 基础异常边界

### 4.2 ADK 管理这些内容的原因

这些内容有一个共同特点：

- 每个 Agent 项目都要写
- 重复度高
- 写错后很难排查
- 不属于具体业务

所以它们应该被 ADK 吞进去，减少外部项目的心智负担。

## 6. 哪些暴露给外部项目

### 5.1 外部项目负责的内容

v0.1.0 中，以下内容应由外部项目负责：

1. 业务工具
   - 例如设备履历查询
   - 例如知识库搜索
   - 例如工单创建

2. 业务提示词
   - 每个 Agent 的系统提示词
   - 每个 Agent 的回复风格与约束

3. provider 选择与配置
   - API Key
   - Model
   - BaseUrl
   - Timeout
   - 通用推理参数
   - 厂商扩展参数

4. 宿主方式
   - 控制台
   - ASP.NET
   - Windows Service
   - 后台任务

5. 业务服务与数据源
   - 数据库
   - HTTP API
   - 领域服务
   - Repository / Query Service

### 5.2 为什么这些不应进 ADK

这些内容带有明显业务或部署环境特征：

- 并不是每个 Agent 都一样
- 变化快
- 一旦塞进 ADK，就会快速污染边界

所以 v0.1.0 要明确克制：

- ADK 不内置设备履历工具
- ADK 不内置 SQL
- ADK 不内置业务 handler
- ADK 不替用户决定模型与提示词

## 7. v0.1.0 不做什么

为了避免过度设计，v0.1.0 明确不做以下能力。

### 6.1 不做复杂 DSL

不做类似下面这种链式框架表达：

```csharp
prompt | model | parser | tools | memory
```

原因：

- 现在还没有稳定的使用习惯
- 抽象过早，容易把简单问题复杂化

### 6.2 不做复杂 Agent 类型体系

不预置：

- `BaseAgent`
- `PlannerAgent`
- `WorkflowAgent`
- `ReActAgent`

原因：

- 当前目标是“简约可用”
- 不是先做一个大框架分类树

### 6.3 不做插件系统

不做：

- 程序集动态加载 tool
- 插件市场
- 自动扫描插件包

原因：

- 这不是当前痛点
- 容易显著增加复杂度

### 6.4 不做持久化 Session Store

v0.1.0 中，`Session` 只管理进程内内存状态，不负责：

- Redis 持久化
- 数据库存档
- 自动恢复会话

原因：

- 当前先把内存会话模型跑顺
- 持久化应该是后续扩展点，而不是 v0.1.0 的默认能力

### 6.5 不做上下文压缩和摘要

暂不做：

- 自动裁剪消息
- 自动总结历史
- 自动窗口压缩

原因：

- 这会引入更多策略问题
- 现在先保证基础会话模型足够清晰

### 6.6 不耦合 ASP.NET Core Web 宿主模型

`crystal-adk` 在 v0.1.0 中是一个基于 .NET 的普通类库，可以运行在 `net6.0` 上，但不直接耦合 ASP.NET Core Web 宿主模型。

这里需要明确区分两件事：

- `net6.0` 是目标运行时版本
- ASP.NET Core 是 Web 应用宿主与框架模型

因此，v0.1.0 不直接依赖以下 Web 宿主相关类型与模式：

- `HttpContext`
- `WebApplication`
- `MapPost`
- `IResult`
- `SSE`

原因：

- ADK 应该是纯运行时 + Session 体验层
- Web 只是宿主形式之一

### 6.7 v0.1.0 事件流范围控制

v0.1.0 对外提供 `StreamEventsAsync(...)`，但事件流范围会刻意收敛。

本期同时做：

- `RunAsync(...)`：非流式，返回最终结果
- `StreamTextAsync(...)`：文本流，只暴露助手文本增量
- `StreamEventsAsync(...)`：Agent 事件流，暴露稳定语义事件

本期事件流重点覆盖：

- `RunStarted`
- `ToolCallStarted`
- `ToolCallCompleted`
- `TextDelta`
- `FinalAnswer`
- `RunCompleted`
- `RunFailed`

本期仍然不做：

- provider 原始 chunk 全量透传
- 厂商级别的底层事件直出
- 过细的内部调度事件

原因：

- 事件命名应该由 ADK 语义层定义，而不是绑定某家协议
- 不同厂商的流式协议差异较大，先做稳定语义层更合理
- v0.1.0 需要的是“好用且可扩展”的事件流，而不是“最底层、最原始”的事件流

## 8. Session 设计结论

关于会话与消息状态，我们在本轮讨论中的结论如下：

1. `Session` 应该管理“会话内消息状态”
2. `Session` 是外部开发者的主要使用入口
3. `Session` 应提供 `RunAsync`
4. `Session` 在 v0.1.0 应提供文本流接口 `StreamTextAsync`
5. `Session` 在 v0.1.0 也应提供 Agent 事件流接口 `StreamEventsAsync`
6. `Session` 必须允许开发者拿出当前对话数组
7. 但对外应优先暴露只读消息列表，而不是可随意修改的内部集合

因此 v0.1.0 推荐的接口方向是：

```csharp
public sealed class AgentSession
{
    public IReadOnlyList<RuntimeMessage> Messages { get; }

    public Task<AgentRunResult> RunAsync(string userInput, CancellationToken cancellationToken = default);

    public IAsyncEnumerable<TextChunk> StreamTextAsync(string userInput, CancellationToken cancellationToken = default);

    public IAsyncEnumerable<AgentEvent> StreamEventsAsync(string userInput, CancellationToken cancellationToken = default);

    public void Reset();
}
```

这个接口表达的核心意图是：

- 用户只管“发一句话”
- ADK 负责把这句话推进成完整的一轮 Agent 交互
- 但用户仍可观察会话状态

## 9. Provider 参数设计结论

关于 Chat Provider 的参数设计，v0.1.0 的结论是：

1. 不能只支持 `ApiKey / Model / BaseUrl / TimeoutMs` 这几个最小字段
2. 也不能直接把每个厂商的全部字段平铺到一个超大 Options 类里
3. 应采用“扁平共性字段 + 厂商扩展字段”的双层模型

### 8.1 为什么采用“扁平共性字段 + 厂商扩展字段”

当前计划兼容的三家协议包括：

- ARK
- Ollama
- ZAI

它们的共性非常明显：

- `model`
- `messages`
- `stream`
- `tools`
- `tool_choice`
- `temperature`
- `top_p`
- `max_tokens` 或等价字段
- `stop`

但差异也很明显：

- ARK 有 `thinking.type`、`reasoning_effort`、`parallel_tool_calls`
- Ollama 有 `options`、`think`、`keep_alive`、`format`
- ZAI 有 `thinking.clear_thinking`、`tool_stream`、`do_sample`、`request_id`、`user_id`

因此如果只保留极少共性字段，开发者会觉得不够用。

但如果把所有厂商字段都硬塞到一个统一类里：

- API 会很臃肿
- 含义会变模糊
- 很多字段在其它厂商上根本不成立

### 8.2 推荐的 Options 结构

v0.1.0 不再额外嵌套 `ChatRequestOptions`，而是将“连接参数”和“通用请求参数”放在同一个 `ProviderOptions` 上，保持开发者配置体验足够直接。

结构上分为两类：

1. 扁平共性字段
   - `ApiKey`
   - `Model`
   - `BaseUrl`
   - `TimeoutMs`
   - `Temperature`
   - `TopP`
   - `MaxOutputTokens`
   - `Stop`
   - `ToolChoice`
   - `ParallelToolCalls`
   - `ResponseFormat`
   - `EnableThinking`

2. 厂商扩展字段
   - `VendorOptions`
   - 类型建议为 `Dictionary<string, object?>`

示意如下：

```csharp
public sealed class OpenAiChatProviderOptions
{
    public string ApiKey { get; set; } = string.Empty;
    public string Model { get; set; } = string.Empty;
    public string BaseUrl { get; set; } = string.Empty;
    public int TimeoutMs { get; set; } = 30000;

    public double? Temperature { get; set; }
    public double? TopP { get; set; }
    public int? MaxOutputTokens { get; set; }
    public IReadOnlyList<string>? Stop { get; set; }
    public string? ToolChoice { get; set; }
    public bool? ParallelToolCalls { get; set; }
    public string? ResponseFormat { get; set; }
    public bool? EnableThinking { get; set; }

    public Dictionary<string, object?> VendorOptions { get; set; } = new();
}
```

这样设计的原因是：

- 外部开发者配置时更顺手
- v0.1.0 更符合“少一层抽象”的目标
- 大部分使用者会把这些都理解为“provider 配置”，没必要强行再套一层

### 8.3 字段映射原则

v0.1.0 建议字段映射遵循以下原则：

1. 优先支持三家都有或语义接近的字段
2. 对命名不同但语义接近的字段做统一抽象
3. 无法统一的字段放入 `VendorOptions`
4. Provider 在发 HTTP JSON 时负责把通用字段转换成厂商字段

例如：

- 通用 `MaxOutputTokens`
  - ARK -> `max_tokens` 或 `max_completion_tokens`
  - Ollama -> `options.num_predict`
  - ZAI -> `max_tokens`

- 通用 `EnableThinking`
  - ARK -> `thinking.type = enabled/disabled`
  - Ollama -> `think = true/false` 或级别字符串
  - ZAI -> `thinking.type = enabled/disabled`

### 8.4 v0.1.0 的克制点

这一版虽然支持 `VendorOptions`，但仍要保持克制：

- 不追求一次性覆盖所有厂商全部字段
- 不做复杂 schema 校验器
- 不做厂商协议自动发现
- 不做完整多模态统一抽象

本期重点仍然是：

- 多轮 messages
- Function Calling
- 非流式 `RunAsync`
- 文本流 `StreamTextAsync`
- Agent 事件流 `StreamEventsAsync`

---

## 10. v0.1.0 的最小成功标准

如果这一版设计落地成功，那么外部项目开发一个新 Agent 时，至少可以做到：

1. 不自己维护 `messages` 数组追加逻辑
2. 不自己维护 tool loop
3. 不自己手动把 tool result 回填到消息中
4. 可以直接 `RunAsync(...)`
5. 可以切换成文本流 `StreamTextAsync(...)`
6. 在需要时可以切换成 Agent 事件流 `StreamEventsAsync(...)`
7. 可以随时读取当前会话消息历史
8. 业务项目只需关心工具、提示词、provider 配置与宿主方式

这就是 `crystal-adk v0.1.0` 设计的目标边界。

## 7. 当前版本已确认的设计结论

截至当前代码状态，v0.1.0 已经确认以下设计结论：

- `AgentSession` 负责会话内消息状态
- 外部项目可以从 `Session.Messages` 取出完整消息数组
- `RunAsync(...)` 是非流式接口
- `StreamTextAsync(...)` 是普通文本流式输出接口
- `StreamEventsAsync(...)` 是 Agent 事件流接口
- 无工具场景不应强迫开发者手写 `ToolHub`
- `AgentHost` 已提供无工具简化重载
- Provider 外部体验统一，内部实现按厂商拆分
